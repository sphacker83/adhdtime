⚠️ 비활성 문서(Deprecated)

이 문서는 “풀데이(8시간) 템플릿” 아이디어를 정리한 참고용 문서입니다.
현재 프로젝트의 `data/validation_rules.json`은 `missions` 개수를 3~6개로 강제하므로,
**이 문서의 ‘20~40개 미션’ 풀데이 템플릿은 현 스키마/검증 계약과 불일치**합니다.

따라서 현 단계에서는 이 문서를 실행 지시/생성 프롬프트로 사용하지 않습니다.
(또한 데이터 생성을 위해 스크립트/코드를 만들거나 조립하는 방식은 금지입니다.)

⸻

docs/full-day-template-generator.md — (비활성) 풀데이(8시간) 템플릿 아이디어 메모

0) 목적

사용자가 “휴일 갓생 계획 만들어줘”, “휴일 힐링 플랜”, “창작 몰입 하루 계획”처럼 입력했을 때 상위로 추천될 수 있는 풀데이(Full-day) 템플릿을 생성한다.
	•	생성 대상: data/templates.json에 추가할 Template 객체들
	•	특징: 480분(8시간) 기준의 “하루 계획” 형태, 수십 개 미션으로 구성
	•	목표: 실제로 따라 할 수 있는 현실성, 기승전결 흐름, 시작/종료 명확, 문장 친절

스키마/필드 제약/타입 정의는 docs/dataset-schemas.md를 단일 기준으로 한다.

⸻

1) 풀데이 템플릿은 “특수 타입”이다

일반 템플릿은 미션 3~6개가 기본이지만, 풀데이 템플릿은 예외로 **수십 개 미션(20~40개 권장)**을 허용한다.
	•	풀데이 템플릿은 clusterKey를 ROUTINE_*_DAY 계열로 분리 운영한다.
	•	추천 엔진이 “하루 계획/휴일 플랜” 입력에서 풀데이 클러스터가 뜨도록, lexicon에 표현을 충분히 연결한다.

⸻

2) 풀데이 템플릿 설계 원칙(절대 규칙)

2.1 “기승전결”을 반드시 가진다
	•	기(시작): 컨디션/상황 확인 + 목표/기준 설정 + 환경 세팅
	•	승(진행): 핵심 활동 블록(집중/창작/학습/외출 등)
	•	전(확장): 보완 활동(정리/운동/정산/소확성취 등)
	•	결(마무리): 정리/내일 준비/완료 체크/짧은 기록

2.2 ‘구체 행동’ 문장만 허용

각 미션은 사용자가 “지금 당장 무엇을 하면 되는지”가 보여야 한다.
	•	동사 + 대상 + 조건/예시(가능하면)

2.3 불친절/가비지 표현 금지

다음과 같은 “메타 문장” 금지:
	•	“종료: 완료 체크 + 다음 액션 기록”
	•	“리마인더”
	•	“정리(최소한)”
	•	“체크하기”만 단독(무엇을 체크하는지 없음)

대신:
	•	“앱에서 완료 버튼 누르기”
	•	“다음에 할 일 1줄 메모(예: …)”
	•	“알림 1개 설정(언제/무엇)”

2.4 시간 합계는 반드시 정확히 맞춘다
	•	sum(missions.estMin) == time.default는 오차 없이 일치해야 한다.
	•	기본은 480분.
	•	변형(3시간/5시간)은 별도 생성 규칙으로 만든다(필요 시).

⸻

3) 풀데이 템플릿의 권장 구성(블록 가이드)

3.1 블록 구성(예시 비율)
	•	준비/리셋: 30~60분
	•	핵심 활동 1: 90~150분
	•	식사/정리: 45~70분
	•	외출/운동/산책: 30~60분
	•	핵심 활동 2: 90~150분
	•	생활 리셋(집안/세탁/정리): 20~50분
	•	마무리(정산/내일 준비): 20~40분

3.2 미션 길이 권장
	•	짧은 실행: 1~6분(준비/정리/체크)
	•	중간 블록: 10~25분(일상 작업)
	•	긴 몰입 블록: 30~45분(창작/학습/집중) — 필요할 때만

⸻

4) 생성 입력 파라미터(LLM에 주는 값)

아래 파라미터로 1개 템플릿을 생성한다.
	•	clusterKey (예: ROUTINE_HOLIDAY_HEALING_DAY)
	•	templateType (대개 narrative, 경우에 따라 process)
	•	titleTheme (예: “휴일 힐링”, “창작 몰입”, “야외 여가”)
	•	concepts[] (예: ROUTINE.HOLIDAY_HEALING, CREATIVE.PROJECT)
	•	contexts[] (예: HOME, MORNING, OUTSIDE)
	•	states[] (보통 비우되, ‘의욕 없음’ 특화 버전은 넣을 수 있음)
	•	targetDefaultMinutes (=480)

⸻

5) 출력 규칙(필수)
	•	출력은 Template JSON 객체 1개만
	•	코드블록/설명/주석 없이 JSON만 출력(데이터 파일에 바로 붙여넣기 용도)
	•	필드는 docs/dataset-schemas.md 스키마를 준수

⸻

6) Codex/LLM 프롬프트(복붙용)

6.1 풀데이 템플릿 1개 생성 프롬프트

너는 ADHD 사용자에게 “휴일 하루 계획(풀데이)”을 제공하는 템플릿을 생성한다.
출력은 docs/dataset-schemas.md의 templates.json 스키마를 따르는 Template JSON 객체 1개다.

[입력 파라미터]
- clusterKey: {CLUSTER_KEY}
- type: {process|narrative|friction}  (풀데이는 보통 narrative)
- title: {사용자에게 보일 제목}
- concepts: {["...","..."]}
- contexts: {["...","..."]}
- states: {[] 또는 ["STATE...."]}
- targetDefaultMinutes: 480
- theme: {예: 힐링/창작/야외/자기계발/친구만남}

[절대 규칙]
1) missions는 “수십 개”(20~40개 권장)로 구성한다.
2) 기승전결 흐름을 반드시 가진다: 시작(세팅) → 핵심활동 → 보완활동 → 마무리(정산/내일준비).
3) 각 mission.action은 “구체 행동”이어야 한다(동사+대상+조건/예시).
4) 금지 표현: "종료:", "리마인더", "다음 액션 기록", "정리(최소한)", "대충", "개기/정리".
5) missions.estMin 합계는 반드시 480과 정확히 일치해야 한다(오차 금지).
6) 템플릿 제목/테마와 무관한 미션은 절대 넣지 말 것.
7) 완료/기록/알림을 말하려면 반드시 구체 행동으로 쓸 것:
   - "앱에서 완료 버튼 누르기"
   - "다음에 할 일 1줄 메모(예: ...)"
   - "알림 1개 설정(언제/무엇)"

[출력]
Template JSON 객체 1개만 출력.

6.2 풀데이 템플릿 “재작성(수정)” 프롬프트

아래 Template JSON은 풀데이 템플릿 품질 규칙을 위반했다.
docs/dataset-schemas.md 스키마는 유지하면서 다음을 만족하도록 수정하라:

- 기승전결 흐름 강화(시작/진행/확장/마무리)
- 가비지/불친절 문장 제거 → 구체 행동 문장으로 교체
- 목표(제목/테마)와 불일치 미션 제거/교체
- sum(estMin)==480 반드시 유지(오차 금지)
- 출력은 수정된 Template JSON 객체 1개만

7) 추천이 잘 뜨게 만드는 연결 지침(데이터 운영)

풀데이 템플릿은 “일반 할 일”과 다르게 추천되어야 한다. 추천이 잘 뜨려면 다음 연결을 지킨다.

7.1 concept/cluster 연결
	•	풀데이 전용 컨셉을 만든다(예: ROUTINE.HOLIDAY_PLAN, ROUTINE.HOLIDAY_HEALING)
	•	concept_to_cluster.json에서 해당 컨셉이 풀데이 클러스터를 우선 포함하도록 매핑한다.

7.2 lexicon 표현 확장(핵심)

다음 입력 표현이 풀데이 컨셉으로 매칭되도록 lexicon을 확장한다.
	•	“휴일 하루 계획”
	•	“갓생 계획”
	•	“풀타임 여가”
	•	“완벽한 하루”
	•	“하루 루틴 짜줘”
	•	“오늘 하루 알차게”
	•	“8시간 플랜”
	•	“하루 통째로”

⸻

8) 생성 작업 권장 세트(바로 쓰는 클러스터 예시)

아래는 풀데이 템플릿으로 특히 효율 좋은 테마들이다. (각 테마당 5~15개씩 생성 권장)
	•	ROUTINE_HOLIDAY_HEALING_DAY (회복/힐링)
	•	ROUTINE_CREATIVE_IMMERSION_DAY (창작 몰입)
	•	ROUTINE_DEEP_GROWTH_DAY (자기계발/학습)
	•	ROUTINE_OUTDOOR_LEISURE_DAY (야외/카페/산책)
	•	ROUTINE_SOCIAL_DAY (사람 만남/관계 회복)
	•	ROUTINE_ERRANDS_DAY (장보기/정리/행정 처리)

⸻

9) 검증(참고)

풀데이 템플릿도 일반 템플릿과 동일하게 검증한다.
	•	sum(estMin)==480 (오차 0)
	•	금지 표현 미포함
	•	제목/테마와 미션 일관성
	•	시작/마무리 존재

검증은 `npm run -s dataset:validate`로 수행한다.
(검증 규칙의 단일 기준은 data/validation_rules.json)

⸻

10) 실행 지시 없음

이 문서는 “아이디어/테마 참고”까지만 제공한다.
풀데이 템플릿을 실제로 도입하려면, 먼저 스키마/검증 계약(`missions` 개수 제한 포함)부터 별도 트랙으로 확장/합의해야 한다.
